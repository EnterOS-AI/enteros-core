# CLI-runtime integration boundary

CLI-based agents integrate through runtime/template-specific boot and session
adapters. Core intentionally does not prescribe one process supervisor,
configuration directory, prompt filename, or inbound-delivery mechanism for all
CLIs.

Current operator setup is generated by
`workspace-server/internal/handlers/external_connection.go` and served through
Canvas **Config → External Connection**. Those server-stamped snippets pin the
reviewed runtime/channel versions, guard one-time credentials, and describe the
runtime's actual push, poll, or channel path.

When adding or changing a CLI runtime:

1. update the runtime-owned adapter and contract tests;
2. update the corresponding workspace template and immutable Core manifest pin;
3. update the server-generated external connection payload where applicable;
4. regenerate the Canvas golden fixture;
5. execute the rendered setup or equivalent semantic harness; and
6. validate registration, heartbeat, inbound delivery, and reply routing.

Do not add LangGraph-specific dependencies or a generic `main.py` flow merely
because older CLI documentation used that shape. The exact template/runtime is
the source of truth.

See [Workspace-runtime boundary](./workspace-runtime.md) and [External Agent
Registration](../guides/external-agent-registration.md).
