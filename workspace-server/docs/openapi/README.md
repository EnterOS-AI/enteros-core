# Workspace-server OpenAPI artifacts

`swagger.yaml` and `swagger.json` are generated by the current swaggo surface.
They cover only the routes that have swag annotations and therefore are not a
complete workspace-server API contract.

The former hand-authored `management.yaml` was removed. It mixed control-plane
and tenant routes, described a retired one-EC2-per-org deployment model, and had
no generator or parity gate tying it to either service. It must not be used to
generate clients or presented as a source of truth.

Current authority is split by service:

- tenant routing and middleware: `workspace-server/internal/router/router.go`;
- tenant request/response behavior: `workspace-server/internal/handlers/`;
- control-plane routes and schemas: the checked
  `molecule-controlplane` repository;
- supported public user flows: the maintained documentation in `docs/`.

When a complete OpenAPI contract is introduced, generate or verify it against
both routers in CI and keep the two authentication domains explicit. Until
then, do not infer unannotated routes from the generated swagger stub.

To refresh the existing annotated stub, use the repository's documented swaggo
generation command and commit both JSON and YAML outputs together.
